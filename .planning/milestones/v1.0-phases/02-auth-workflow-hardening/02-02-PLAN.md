# 02-02 Plan: Lock Authenticated Landing And CI Behavior

## Objective

<objective>
Turn current authenticated inspection into committed Playwright coverage:
storage state must reach `/applications`, the sign-in screen must stay hidden,
the `Applications` page landmarks must be visible, and CI/docs must describe the
skip-absent/fail-invalid auth behavior.
</objective>

## Context

- `.planning/phases/02-auth-workflow-hardening/02-CONTEXT.md`
- `.planning/phases/02-auth-workflow-hardening/02-RESEARCH.md`
- `.planning/phases/02-auth-workflow-hardening/02-PATTERNS.md`
- `.planning/phases/02-auth-workflow-hardening/02-VALIDATION.md`
- `.planning/phases/01-automation-foundation/01-VERIFICATION.md`

## Threat Model

| ID         | Threat                                                                  | Severity | Mitigation                                                                                   |
| ---------- | ----------------------------------------------------------------------- | -------- | -------------------------------------------------------------------------------------------- |
| T-02-02-01 | Authenticated smoke passes without proving the stable app landing page. | high     | Assert URL `/applications`, title, heading, and stable navigation/form landmarks.            |
| T-02-02-02 | CI silently hides invalid provided auth state.                          | high     | Keep auth job skipped only when secret absent; malformed/expired provided state fails setup. |
| T-02-02-03 | Tests overfit to user-specific profile text.                            | medium   | Assert app landmarks, not account name or email.                                             |
| T-02-02-04 | Phase 2 drifts into Phase 3 workflow coverage.                          | medium   | Limit assertions to authenticated landing/work-area baseline; defer case/document flow.      |
