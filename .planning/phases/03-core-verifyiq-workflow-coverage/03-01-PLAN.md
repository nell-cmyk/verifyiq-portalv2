# 03-01 Plan: Add Application Workflow Foundation

## Objective

<objective>
Create the Phase 3 executable foundation for Add Application coverage: add a
synthetic upload fixture, add narrow workflow helpers, and add one authenticated
happy-path Playwright test that reaches `/applications/new`, submits a sandbox
application with generated `AUTOMATION` data, and proves the created record is
visible or opened.
</objective>

## Decision Coverage

- D-01, D-02, D-03, D-04: Add Application is the primary path, full submission
  is allowed in sandbox, and post-submit must prove the created record is
  visible or opened.
- D-05, D-09, D-10: use the default/first stable visible document type and a
  committed synthetic fixture; assert visible accepted row state.
- D-07, D-16, D-17, D-18, D-19: generated safe defaults use `AUTOMATION` plus a
  timestamp/run id; helper shape is based on actual form complexity.
- D-11, D-12: run in default authenticated CI and Desktop Chrome.
- D-13, D-14, D-15: include form inventory diagnostics, role/label-first
  locators, test ids for ambiguous controls, and narrow helpers only.

## Threat Model

<threat_model>

| ID         | Threat                                                                                                          | Severity | Mitigation                                                                                                                            |
| ---------- | --------------------------------------------------------------------------------------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| T-03-01-01 | A committed upload fixture contains real personal, financial, credential, cookie, token, or storage-state data. | high     | Use only a hand-written synthetic fixture containing `SYNTHETIC VERIFYIQ AUTOMATION FIXTURE` and no real account or financial values. |
| T-03-01-02 | Add Application submissions create durable sandbox records that are hard to identify.                           | high     | Generate applicant names with `AUTOMATION` plus timestamp or run id before every submit.                                              |
| T-03-01-03 | Duplicated `Add Document` controls cause the test to upload to the wrong section.                               | medium   | Use scoped locators or stable test ids such as `add-primary-doc-btn`, `doc-type-select-1`, and `file-input`.                          |
| T-03-01-04 | Failure diagnostics leak secrets from env, storage state, cookies, or screenshots.                              | high     | Attach only visible form inventory, labels, roles, placeholders, button text, and safe test ids.                                      |
| T-03-01-05 | The test passes on a toast without proving the created record exists.                                           | high     | Require generated applicant name visibility or an opened created-record route after submit.                                           |

</threat_model>
