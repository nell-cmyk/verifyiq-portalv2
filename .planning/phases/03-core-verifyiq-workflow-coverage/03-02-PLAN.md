# 03-02 Plan: Expand Workflow Matrix And Document Test Data Behavior

## Objective

<objective>
Expand Phase 3 from the first Add Application happy path into bounded visible
workflow coverage: exercise stable document type variants, add one deterministic
validation assertion, allow workflow-specific timeout where needed, and align
README/planning docs with durable sandbox test-data behavior.
</objective>

## Decision Coverage

- D-05, D-18, D-19: the planner/executor may choose optional field handling and
  helper shape based on form complexity, but must stay stable.
- D-06, D-08, D-10: cleanup remains visible-UI-only, API setup is allowed only
  for prerequisites, and upload assertions use stable accepted state.
- D-20, D-21, D-22, D-23: full visible workflow mutations are allowed in
  sandbox, processing assertions are required only when fast/stable, and docs
  must warn about durable sandbox records.
- D-24, D-25, D-26, D-27: plan for a bounded matrix over stable visible
  variants, one deterministic validation assertion, and longer workflow-specific
  timeout only when async processing is core to value.

## Threat Model

<threat_model>

| ID         | Threat                                                                                                    | Severity | Mitigation                                                                                                                 |
| ---------- | --------------------------------------------------------------------------------------------------------- | -------- | -------------------------------------------------------------------------------------------------------------------------- |
| T-03-02-01 | Matrix tests create excessive sandbox records or make CI too slow.                                        | high     | Bound matrix to visible stable options and use one generated `AUTOMATION` record per selected variant.                     |
| T-03-02-02 | Validation coverage becomes brittle by depending on styling or hidden implementation details.             | medium   | Assert only the visible deterministic message `Please enter the applicant name.` on `/applications/new`.                   |
| T-03-02-03 | Async processing checks create flakes.                                                                    | medium   | Use a workflow-specific timeout only for stable visible results; otherwise document and stop at created-record visibility. |
| T-03-02-04 | Docs fail to warn humans and CI owners that authenticated workflow tests can create durable sandbox data. | high     | Update README and planning docs with explicit durable data and best-effort cleanup language.                               |
| T-03-02-05 | Expanded assertions overfit to account-specific data or hidden options.                                   | medium   | Limit matrix to stable visible document type labels and generated applicant names.                                         |

</threat_model>
