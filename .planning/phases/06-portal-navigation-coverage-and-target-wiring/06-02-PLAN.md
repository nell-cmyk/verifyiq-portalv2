## Decision Coverage

- D-01: create one focused portal navigation spec for page availability.
- D-05: assert URL, heading, navigation affordance, and primary page surface.
- D-06: choose stable rich assertions after live inspection only.
- D-07: assert page shell and stable containers instead of account data.
- D-08: reuse `collectPageErrors` in each portal page test.
- D-09: prove visible nav and direct route fallback.
- D-10: derive direct routes from live nav hrefs.
- D-11: fail clearly when a required nav link is hidden or absent.

## Threat Model

<threat_model>

| ID      | Threat                                                             | Severity | Mitigation                                                                                                                      |
| ------- | ------------------------------------------------------------------ | -------- | ------------------------------------------------------------------------------------------------------------------------------- |
| T-06-06 | Tests mutate portal records before the safety harness exists.      | high     | Portal navigation spec must not click create, update, delete, invite, save, or cleanup controls.                                |
| T-06-07 | Direct-route checks guess route slugs.                             | high     | Read each route from visible nav link `href` during the test and reuse that href for direct navigation.                         |
| T-06-08 | Missing role-permission nav links silently skip required coverage. | high     | Throw an explicit error naming the missing label and target.                                                                    |
| T-06-09 | Assertions depend on account-specific rows or existing records.    | medium   | Assert headings, nav, `main` or stable shell containers, and optional stable controls only.                                     |
| T-06-10 | Page or console errors are missed.                                 | high     | Call `collectPageErrors(page)` before navigation and `expectNoErrors(testInfo)` after assertions in every test.                 |
| T-06-11 | Auth-state failures are misreported as portal regressions.         | medium   | Keep authenticated tests on `authenticated-chromium` setup and record `npm run auth:record` recovery if storage state is stale. |

</threat_model>

  <action>
    Create `tests/support/portal-navigation.ts` using TypeScript and
    `@playwright/test` locators only. Do not add dependencies.

    Export a readonly `portalAreas` array with exactly these definitions:

    - target: `applications`, tag: `@portal:applications`, label:
      `Applications`, heading regex: `/^Applications$/i`
    - target: `activity`, tag: `@portal:activity`, label: `Activity`, heading
      regex: `/^Activity$/i`
    - target: `audit-logs`, tag: `@portal:audit-logs`, label: `Audit Logs`,
      heading regex: `/^Audit Logs$/i`
    - target: `users`, tag: `@portal:users`, label: `Users`, heading regex:
      `/^Users$/i`
    - target: `roles`, tag: `@portal:roles`, label: `Roles`, heading regex:
      `/^Roles$/i`

    Export these helper functions:

    - `getRequiredPortalNavLink(page, area)`
    - `resolvePortalHref(page, area)`
    - `expectPortalAreaShell(page, area, expectedUrlPattern)`
    - `expectPortalAreaReachable(page, area)`

    `getRequiredPortalNavLink` must locate
    `page.getByRole("link", { name: new RegExp(`^${area.label}$`, "i") })`.
    If no link is visible, throw:

    `Required portal navigation link "<label>" for target "<target>" was not visible.`

    `resolvePortalHref` must read the visible link's `href`, resolve it with
    `new URL(href, page.url())`, and return a path string including pathname,
    search, and hash. Do not use a static route map for Activity, Audit Logs,
    Users, or Roles.

    `expectPortalAreaShell` must assert:

    - `expect(page).toHaveURL(expectedUrlPattern)`
    - the area heading is visible
    - the area nav link is visible
    - a stable primary surface is visible using `page.locator("main, [role='main']").first()`
    - if live inspection exposes a stable active nav marker, assert it using the
      exact observed signal; otherwise keep visible nav plus URL as the active
      affordance and record the absence of a separate active marker in the
      implementation summary.

    `expectPortalAreaReachable` must:

    1. Resolve the area href from the Applications nav.
    2. Click the visible link.
    3. Assert the area shell after visible navigation.
    4. Navigate directly to the previously resolved href path.
    5. Assert the same area shell after direct navigation.

  </action>
