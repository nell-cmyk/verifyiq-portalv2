# 02-01 Plan: Harden Auth-State Setup And Diagnostics

## Objective

<objective>
Make auth setup deterministic and secret-safe: env storage state wins over local
state, `VERIFYIQ_FORCE_LOGIN=1` bypasses only local state, every reused state is
validated before authenticated tests run, and auth failures identify the recovery
path without leaking credentials or serialized storage state.
</objective>

## Context

- `.planning/phases/02-auth-workflow-hardening/02-CONTEXT.md`
- `.planning/phases/02-auth-workflow-hardening/02-RESEARCH.md`
- `.planning/phases/02-auth-workflow-hardening/02-PATTERNS.md`
- `.planning/phases/02-auth-workflow-hardening/02-VALIDATION.md`
- `docs/ai-development-workflow.md`

## Threat Model

| ID         | Threat                                                                    | Severity | Mitigation                                                                                      |
| ---------- | ------------------------------------------------------------------------- | -------- | ----------------------------------------------------------------------------------------------- |
| T-02-01-01 | Local auth file masks an explicit CI/shell storage-state override.        | high     | Enforce env-first precedence in setup and cover it with grep-verifiable code structure.         |
| T-02-01-02 | Expired storage state passes setup and fails later with vague app errors. | high     | Validate staged/reused storage state in a fresh context before setup returns.                   |
| T-02-01-03 | Diagnostics leak credentials, cookies, tokens, or storage-state JSON.     | high     | Use fixed messages that name variables and commands but never values.                           |
| T-02-01-04 | CAPTCHA pressure leads to bypass attempts.                                | medium   | Preserve manual `auth:record` and storage-state guidance; do not add solver or bypass behavior. |
| T-02-01-05 | Claude setup failure gets mistaken for fallback.                          | medium   | Run the documented Claude Opus preflight before execution and stop on setup failure.            |

  <action>
    Update `tests/auth.setup.ts` to import helpers from
    `tests/support/auth-state.ts`.

    Replace the current branch order with exactly this precedence:

    1. Always create the auth directory with `await mkdir(dirname(authFile), { recursive: true });`
    2. If `VERIFYIQ_STORAGE_STATE_JSON` is set, call `writeStorageStateJson(storageStateJson)`, then `validateStoredAuthState(browser, "VERIFYIQ_STORAGE_STATE_JSON")`, then return.
    3. If `VERIFYIQ_STORAGE_STATE_PATH` is set, call `copyStorageStateFile(storageStatePath)`, then `validateStoredAuthState(browser, "VERIFYIQ_STORAGE_STATE_PATH")`, then return.
    4. If `VERIFYIQ_FORCE_LOGIN !== "1"` and `fileExists(authFile)` is true, call `validateStoredAuthState(browser, "playwright/.auth/user.json")`, then return.
    5. Otherwise require username/password, call `signIn(page, username, password)`, save `page.context().storageState({ path: authFile })`, then return.

    Use `const browser = page.context().browser();` and throw a fixed error if
    it is null:

    `Playwright browser is unavailable for VerifyIQ auth-state validation.`

    `VERIFYIQ_FORCE_LOGIN=1` must only skip branch 4. It must not skip branches
    2 or 3.

  </action>
  <verify>

    - `rg -n 'VERIFYIQ_STORAGE_STATE_PATH' tests/auth.setup.ts`

    - `rg -n 'validateStoredAuthState\\(browser, "VERIFYIQ_STORAGE_STATE_JSON"\\)' tests/auth.setup.ts`
    - `rg -n 'validateStoredAuthState\\(browser, "VERIFYIQ_STORAGE_STATE_PATH"\\)' tests/auth.setup.ts`
